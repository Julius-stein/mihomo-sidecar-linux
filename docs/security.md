# Security

本项目运行在“共享服务器 + root 级网络规则 + 用户级代理需求”这个组合场景里，因此安全边界必须说清楚。

## 当前安全模型

当前仓库的安全前提是：

- Mihomo 作为 systemd 服务运行
- 路由与防火墙规则由高权限脚本安装
- 只有特定 GID 的进程会被送入 sidecar 路径
- Mihomo controller 默认只监听 `127.0.0.1:9091`
- 节点切换通过 `MIHOMO_API_SECRET` 进行鉴权

这意味着本项目不是“无特权用户随手即可安装”的工具，而是需要管理员或受控运维流程参与的网络工具。

## 最重要的安全建议

## 1. controller 只绑定本地回环

当前实现默认使用：

- `external-controller: 127.0.0.1:9091`

这个约束应当保留。不要把 controller 暴露到公网，也不要绑定到共享网卡地址。

## 2. 不要把真实 secret 提交到仓库

这些内容都不应进入 Git：

- `MIHOMO_API_SECRET`
- 带真实 `secret:` 的运行时 `config.yaml`
- 节点订阅 URL
- 代理节点密码、token、证书材料

推荐做法：

- 仓库只保留模板文件
- 实际运行配置放到受控目录
- 使用最小可读权限，例如 `0640` 或更严格

## 3. 共享服务器上谨慎管理 sidecar 组成员

当前进程级代理是通过 `--gid-owner` 实现的。

这意味着：

- 任何能以目标 GID 运行命令的人，都能走 sidecar 路由
- 组成员范围本身就是一个权限边界

因此应当：

- 谨慎把用户加入 `sidecar` 组
- 明确 sidecar 组的用途
- 在审计要求高的环境中保留组成员变更记录

## 4. 防止多实例资源冲突

在共享服务器上，安全问题并不只来自“被攻击”，也来自“被误配置”。

当前仓库默认值包括：

- `fwmark=0x2`
- route table `101`
- priority `1001`
- DNS 端口 `1054`
- controller 端口 `9091`
- fake-ip-range `198.19.0.1/16`
- TUN IPv4 `198.19.0.1/30`

如果多个实例直接复用这些值，可能出现：

- 规则互相覆盖
- 流量错误命中别的实例
- fake-ip 或 TUN 子网冲突
- 排障信息混乱

在共享环境中，应该把这些资源看作“需要分配与登记的命名空间”。

## 5. 注意日志泄漏

排障时常会执行：

- `journalctl`
- `systemctl status`
- controller API 调试命令

这些输出里可能含有：

- 节点名
- 服务器域名
- 本地路径
- 错误上下文

分享日志前应先清理敏感字段。

## 当前已知风险点

以下风险来自当前仓库真实存在的实现方式：

1. 核心参数硬编码，易导致“服务配置已改、规则脚本未同步”。
2. 当前没有统一安装器，手工部署更容易留下权限或路径错误。
3. 当前没有自动资源探测，默认值在多实例场景下冲突概率高。
4. 当前只实现 GID sidecar 模式，尚未定义 UID 透明代理的安全边界。

## 部署建议

在进入生产或公开共享环境前，建议至少做到：

1. controller 仅监听本地回环。
2. 运行时配置与 secret 不进入 Git 仓库。
3. 运行目录权限最小化。
4. sidecar 组成员最小化。
5. 为每个实例分配独立的 fake-ip-range、TUN 子网、mark、table、priority 和端口。
6. 把 systemd unit、配置文件和规则脚本的实际版本一起纳入变更管理。
